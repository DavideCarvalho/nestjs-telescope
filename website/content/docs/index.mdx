Welcome to the **nestjs-telescope** documentation.

`@dudousxd/nestjs-telescope` is an application observability console for NestJS. A set of watchers capture what happens *inside* a request — every query, queued job, sent email, cache hit, and thrown exception — correlate it all under one batch via `AsyncLocalStorage`, store it through a pluggable provider, and expose it through a headless API plus an optional dashboard. It's the missing Telescope for NestJS, designed framework-idiomatically rather than ported.

Head to the [Getting Started](/docs/getting-started) guide to mount the dashboard in under five minutes, or use the sidebar to explore concepts, packages, and the dashboard tour.

## What you get

- **Correlated capture.** The unit of value is the *batch* — one request and everything it caused. Open an entry and see the exact queries, jobs, exceptions, and mails it produced, in capture order.
- **A Horizon-style live queue console.** Browse BullMQ and SQS queues in real time — depth per state, individual jobs, and (behind a second default-deny gate) retry / remove / promote / redrive.
- **A Pulse-style health dashboard.** Per-type entry counts, the slowest entries, the most frequent exceptions, N+1 occurrences, and a throughput sparkline — all rolled up from captured entries.
- **Off the response path.** Request capture fires *after* the response is flushed; query capture is ~microseconds. Telescope is built so that it never slows the app it observes — and it surfaces its own cost on a `/health` card.
- **Error alerting.** A genuinely *new* exception family pages you — Slack (Block Kit, with route/user and a deep link), a raw webhook, or your own sink — alongside rate, slow-route, and dropped-entry rules.
- **AI diagnosis.** One click turns an exception into a probable-cause / suggested-fix report from its stack, route, and the SQL that just ran. In auto-mode it enriches the Slack alert. Bedrock, OpenAI, or any Vercel AI SDK model.
- **Frontend error reporting.** Point a browser error handler at a public endpoint and `client_exception` reports flow through the same pipeline — family hashing, alerts, archive, the dashboard's Client errors tab.
- **Archive before prune.** Per-type retention keeps exceptions a week and requests an hour; an `archive.sink` ships doomed entries to S3 (or any cold store) before the pruner deletes them.
- **MCP for coding agents.** An optional Model Context Protocol server lets Claude Code, Cursor, or any MCP client debug straight from the captured data — *"why is `POST /checkout` slow?"* pulls the batch waterfall with every query.
- **Safe under load.** Capture stays off the response path, and an overload guard pauses recording when the event loop is lagging — so a telescope under pressure never amplifies an incident.

## Why

Backend developers reach for Telescope in two moments: locally, debugging *"why did this request do 47 queries?"*, and in production, where an admin needs to inspect a failing flow without SSHing into a pod to grep logs. You install core plus one adapter, add one module import, and within five minutes you're clicking through a request — in dev with full detail, in prod with redaction and a gate.

It complements metrics and tracing; it doesn't replace them. This is about *correlated entries you click into*, not time-series aggregates.
